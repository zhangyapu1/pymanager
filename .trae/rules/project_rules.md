# pymanager 项目编码规则

## 行为准则：减少常见 LLM 编码错误

### 1. 三思而后行
不要假设。不要隐藏疑惑。明确权衡。

在实现之前：
- 明确陈述你的假设。如果不确定，请问。
- 如果存在多种解释，请先呈现——不要静默选择。
- 如果存在更简单的方法，请说出来。有理由时要勇于反驳。
- 如果某些事情不清楚，请停止。说出什么是不清楚的。请问。

### 2. 简单优先
最小化解决问题的代码。不要写投机取巧的代码。
- 不要添加需求之外的功能。
- 不要为一次性使用的代码创建抽象。
- 不要添加未被要求的"灵活性"或"可配置性"。
- 不要为不可能的场景添加错误处理。
- 如果你写了 200 行而它可以是 50 行，请重写。
- 问自己："高级工程师会说这太复杂了吗？"如果是，请简化。

### 3. 精确修改
只触碰必须修改的地方。清理你自己的烂摊子。

编辑现有代码时：
- 不要"改进"相邻的代码、注释或格式。
- 不要重构没有损坏的东西。
- 匹配现有风格，即使你会用不同的方式做。
- 如果你注意到不相关的死代码，请提及——不要删除它。

当你的更改创建了孤立代码时：
- 移除因你的更改而变得未使用的导入/变量/函数。
- 不要移除预先存在的死代码，除非被要求。

**测试标准**：每一行更改都应该直接追溯到用户的请求。

### 4. 目标驱动执行

定义成功标准。循环验证直到完成。

将任务转化为可验证的目标：
- "添加验证" → "为无效输入编写测试，然后让它们通过"
- "修复 bug" → "编写一个能复现它的测试，然后让它通过"
- "重构 X" → "确保之前和之后测试都通过"

对于多步骤任务，请简述计划：
```
1. [步骤] → 验证：[检查]
2. [步骤] → 验证：[检查]
3. [步骤] → 验证：[检查]
```

强有力的成功标准让你能够独立循环。软弱的标准（"让它工作"）需要不断的澄清。

## 项目特定规则

### 代码风格
- 使用中文注释和文档字符串
- 变量和函数命名使用英文
- 匹配项目现有风格

### Git 提交规范
- 提交信息使用中文，简明描述更改内容
- 每次提交尽量只包含一个功能或修复
- 推送前检查变更内容

### 依赖管理
- 只添加必要的第三方库
- 更新版本号时同步更新 config.py 中的 CURRENT_VERSION
