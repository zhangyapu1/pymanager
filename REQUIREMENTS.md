# pymanager 项目需求文档

> 本文档用于记录 pymanager 的待实现功能建议及同类优秀功能参考。

## 一、项目定位

pymanager 是一个基于 Tkinter 的 Python 脚本管理工具，帮助用户：

- 保存和管理本地 Python 脚本
- 快速运行脚本
- 管理脚本依赖

## 二、待实现功能

### 2.1 核心功能增强

当前无待实现的高优先级核心功能。

### 2.2 脚本运行增强

| 功能               | 描述                          | 复杂度 | 备注                                         |
| ---------------- | --------------------------- | --- | ------------------------------------------ |
| **运行参数**         | 支持为脚本设置默认运行参数               | 中   | <br />                                     |
| **运行前参数输入框**     | 运行前弹窗让用户填写参数（如文件路径、API Key） | 低   | 比默认参数更常见的需求，建议与"运行参数"配合实现                  |
| **Python 版本选择**  | 支持选择不同的 Python 解释器运行脚本      | 中   | <br />                                     |
| **运行历史**         | 保存脚本的运行历史记录（包括输出）           | 中   | 可与"日志查看器"合并为"统一输出面板"                       |
| **后台运行**         | 支持将脚本作为后台进程运行               | 高   | <br />                                     |
| **定时运行（简单版）**    | 支持简单的每分钟/小时定时执行             | 低   | 使用 `threading.Timer` 实现，满足"每隔几小时运行一次"的常见需求 |
| **定时运行（cron 版）** | 支持完整的 cron 表达式定时执行          | 高   | 后期升级，需引入 cron 解析库                          |
| **运行统计**         | 记录每个脚本的运行次数、运行时长            | 中   | <br />                                     |

### 2.3 依赖管理增强

| 功能          | 描述                             | 复杂度 | 备注     |
| ----------- | ------------------------------ | --- | ------ |
| **依赖缓存**    | 缓存已解析的依赖，避免重复解析                | 低   | <br /> |
| **虚拟环境支持**  | 支持为不同脚本创建独立的虚拟环境               | 高   | <br /> |
| **依赖导出/导入** | 导出/导入脚本的依赖列表（requirements.txt） | 低   | <br /> |
| **依赖版本显示**  | 显示已安装依赖的版本号                    | 低   | <br /> |

### 2.4 数据管理

| 功能           | 描述                               | 复杂度 | 备注                                   |
| ------------ | -------------------------------- | --- | ------------------------------------ |
| **数据导出/备份**  | 导出脚本列表和配置为 JSON/YAML             | 低   | <br />                               |
| **脚本导出单一文件** | 将脚本打包成一个 `.pymanager` 文件，方便分享    | 中   | 目前"数据导出/备份"只导出列表，脚本本体还在原路径           |
| **数据导入**     | 从其他工具导入脚本列表                      | 中   | <br />                               |
| **脚本修改历史**   | 每次保存自动备份到 `backups/` 目录          | 低   | 用户体验好，实现简单（复制 + 时间戳命名）               |
| **云同步**      | 支持将配置同步到云端（GitHub Gist / WebDAV） | 高   | 优先考虑 GitHub Gist 或 WebDAV，比自建云端服务成本低 |

### 2.5 用户体验

| 功能           | 描述                             | 复杂度 | 备注                                                            |
| ------------ | ------------------------------ | --- | ------------------------------------------------------------- |
| **主题配色文件支持** | 支持通过配置文件（如 `color.ini`）自定义界面配色 | 低   | 先于深色模式实现，让用户自己改配色文件，复杂度更低                                     |
| **深色模式**     | 支持深色/浅色主题切换                    | 中   | Tkinter 需大量修改样式，建议在主题配色文件基础上实现                                |
| **主题自定义**    | 支持自定义界面颜色、字体                   | 高   | <br />                                                        |
| **最小化到托盘**   | 支持最小化到系统托盘                     | 高   | Tkinter 原生不支持，需引入 `pystray` 库；Windows 下需注意图标格式 (.ico)；跨平台处理复杂 |
| **启动时检查更新**  | 启动时自动检查更新（可选）                  | 低   | 部分已实现：启动时显示版本对比信息，但未自动提示更新                                    |
| **运行通知**     | 脚本运行完成后发送系统通知                  | 低   | <br />                                                        |

### 2.6 高级功能

| 功能         | 描述                    | 复杂度 | 备注                             |

## 四、推荐优先实现功能

基于开发成本和用户价值，建议按以下优先级实现：

### 第一阶段（增强功能）

1. **主题配色文件支持** - 先实现 `color.ini` 配色方案，降低深色模式开发成本
2. **脚本修改历史** - 自动备份，防止误修改
3. **依赖版本显示** - 便于依赖管理
4. **运行前参数输入框** - 提升运行灵活性

### 第二阶段（长期）

1. **深色模式** - 在主题配色文件基础上实现
2. **脚本模板** - 快速创建新脚本
3. **定时运行（简单版）** - 自动化脚本执行
4. **云同步** - 跨设备同步配置

## 五、技术债务

当前无未解决的技术债务。已解决项：

| 问题                          | 解决方案                                                                                               |
| --------------------------- | -------------------------------------------------------------------------------------------------- |
| 代码耦合                        | 已通过 AppContext Protocol + UICallbackProtocol + GroupManagerInterface + ScriptCollectionProtocol 解耦 |
| UI 框架老旧                     | 已引入 `ttkbootstrap`（cosmo 主题），组件使用 `bootstyle` 样式                                                   |
| `edit_content.py` UI 混合     | 已提取编辑器 UI 到独立 `ui_editor.py`（`EditorWindow` 类）                                                     |
| `group_manager.py` 持有 combo | 已将 `create_group_widgets()` 移至 `ui_builder.py`，GroupManager 仅提供数据接口                                |
| Python 2 遗留模块兼容性            | 自动检测并创建兼容垫片（`PY2_SHIM_CONTENT`），覆盖 20+ 个 Python 2 模块                                               |
| 包名冲突（docx/python-docx）      | `PACKAGE_CONFLICTS` 自动检测旧版包并替换为正确版本                                                                |
| 依赖检查无日志                     | `check_deps.py` 和 `run_selected.py` 依赖检查操作写入日志文件                                                   |
| `__pycache__` 磁盘写入          | `sys.dont_write_bytecode = True` 禁用字节码缓存                                                           |
| main.pyw 转发方法过多             | 已移除 16 个纯转发方法，各模块直接访问 ctx 组件                                                                       |
| scripts 列表直接操作              | 已封装为 ScriptCollection 类，提供 add/remove/find\_by\_path/update 接口                                     |

## 六、非功能性需求

| 需求   | 描述                     | 目标                                                    |
| ---- | ---------------------- | ----------------------------------------------------- |
| 启动速度 | 随着脚本数量增加，搜索和加载速度不能明显下降 | 100 个脚本内启动时间 < 2 秒                                    |
| 安全性  | 运行用户脚本时需考虑安全隔离         | 避免 `shell=True` 防止命令注入（已实现）                           |
| 兼容性  | 明确支持的 Python 版本范围      | Python 3.8+；Windows 为主要支持平台（Token 加密依赖 Windows DPAPI） |
| 可移植性 | 配置和脚本数据应能在不同机器间迁移      | 路径使用相对路径（已实现），配置使用 JSON 格式                            |

## 七、开发路线图

### v1.4（已发布 - 依赖管理增强 + 易用性）

- Python 2 兼容性自动修复（兼容垫片系统）
- 包冲突自动替换（docx → python-docx）
- 传递依赖检测增强（三层自动修复）
- 依赖检查日志记录
- 禁用 `__pycache__` 字节码缓存
- 脚本收藏/置顶（右键收藏，收藏脚本置顶显示 ⭐）
- 脚本图标（右键设置图标，20 个内置 emoji 图标）
- 批量操作（Ctrl/Shift 多选，右键批量删除/移动/导出）
- 代码深度解耦（ScriptCollection + Protocol 体系）

### v1.5（已发布 - 搜索与排序）

- 脚本搜索/过滤（列表上方搜索框，实时模糊匹配，支持按名称和路径过滤）
- 最近运行排序（脚本列表按 收藏→最近运行→其他 排序，最近运行脚本按时间倒序，🕒 标记）
- 运行时间戳记录（`recent_runs` 模块，自动清理超过 50 条的旧记录）

### v1.6（已发布 - 脚本市场）

- 脚本市场：GitHub 仓库浏览/搜索、项目下载（整个项目或单个文件/文件夹）、下载进度条
- README 翻译：有道翻译/百度翻译/腾讯翻译君，分段翻译逐段显示
- AI 项目分析：智谱AI / 通义千问，API Key 加密存储，手动选择 AI 自动设为默认
- UI 改进：统一按钮配色、统一窗口底色、三栏可调布局（PanedWindow）
- Bug 修复：scan_data_directory TypeError、GitHub API 403、翻译卡住、TclError、README 预览

### v1.6.1（已发布 - 更新机制增强）

- 清单对比清理：manifest.json + manifest_generator.py，更新时自动删除废弃文件、清理空目录
- GitHub Release 自动发布：create_github_release() 函数，通过 GitHub API 自动创建 Release
- 白名单保护：data/、config/、logs/、backups/ 等用户数据目录更新时永不删除
- Bug 修复：scan_data_directory 返回 None 导致 TypeError、AI 面板按钮不可见

### v1.6.4（已发布 - 更新清理修复）

- 修复更新时强制清理未执行：强制清理逻辑提前到函数开头，确保无论有无 manifest 都会执行

### v1.6.3（已发布 - 更新机制修复）

- 修复下载更新后空目录未清理：循环清理嵌套空目录
- 修复下载更新后未自动重启：重启脚本添加 taskkill /f /pid 强制终止旧进程
- .exe 更新增加文件复制重试机制（最多5次）
- manifest.json 移入 config/ 目录
- 更新时强制清理 .trae/、tests/、.gitignore、REQUIREMENTS.md、根目录 manifest.json
- manifest 生成排除 .trae/、tests/、.gitignore、REQUIREMENTS.md

### v1.6.2（已发布 - 文档改进）

- 为全部 34 个模块添加详细 docstring：功能描述、函数签名、参数说明、数据结构、实现细节、依赖关系
- 修复 utils.py docstring 中三单引号导致语法错误的问题

### v1.7（易用性）

- 主题配色文件支持
- 脚本修改历史（自动备份）

### v1.8（增强）

- 运行前参数输入框
- 深色模式
- 脚本模板
- 定时运行（简单版）

### v2.0（高级）

- 云同步（GitHub Gist / WebDAV）
- 插件系统
- 定时运行（cron 版）
- 虚拟环境支持
